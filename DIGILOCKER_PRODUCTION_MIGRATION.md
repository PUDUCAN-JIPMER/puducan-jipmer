# DigiLocker & ABHA Production Migration Guide
### PuduCan Healthcare Platform

This guide outlines the step-by-step process for transitioning the patient identity verification layer from **Development/Mock Mode** to a fully live **Production-Grade DigiLocker Flow**. 

Follow these steps once you receive the official DigiLocker partner credentials and integration keys from your mentor/NIC portal.

---

## 1. Configure Production Environment Variables

Add or update the following keys in your private environment file (`.env.local` for local development, or your production cloud environment setup):

```bash
# ── PRODUCTION DIGILOCKER OAUTH CREDENTIALS ──────────────────────────────────
# Set this to 'https://api.digitallocker.gov.in' for production,
# or 'https://sandbox.digitallocker.gov.in' for the NIC sandbox environment.
DIGILOCKER_BASE_URL="https://api.digitallocker.gov.in"

# Your production OAuth Client credentials provided by the DigiLocker Portal
DIGILOCKER_CLIENT_ID="your_real_prod_client_id"
DIGILOCKER_CLIENT_SECRET="your_real_prod_client_secret"

# The redirect URI registered in your DigiLocker partner console
# Note: This MUST match the domain of your deployed application
DIGILOCKER_REDIRECT_URI="https://yourdomain.org/api/digilocker/callback"

# ── AES-256-GCM AADHAAR ENCRYPTION KEY ───────────────────────────────────────
# Generate a cryptographically secure 32-byte (64 hex characters) key.
# Run in your terminal to generate: openssl rand -hex 32
DIGILOCKER_AADHAAR_KEY="a_secure_64_character_hex_encryption_key_here"

# ── CLIENT-SIDE FEATURE FLAGS ────────────────────────────────────────────────
# Set to true to activate the real popup-based server OAuth flow in the UI.
# If set to false, PuduCan will continue to use the OTP-based dev simulation.
NEXT_PUBLIC_DIGILOCKER_ENABLED=true
```

## 2. Code Block Cleanups & Deletions

Once the production keys are active, you can clean up the simulated fallback code and dev indicators from the codebase.

### A. Remove OTP Display Card in the Verification Modal
During development, the OTP is generated by the server and displayed inline within an informational alert so that developers do not need an SMS gateway. In production, this can be safely removed.

* **File:** `components/verification/VerificationModal.tsx`
* **Block to Delete:** Look for the alert block containing the message `Dev Mode Simulation OTP` (around line 200–250) and remove it:
  ```tsx
  {/* REMOVE THIS WHOLE BLOCK */}
  {process.env.NODE_ENV !== 'production' && otpCode && (
    <Alert className="bg-blue-50 border-blue-200 ...">
      <InfoIcon className="h-4 w-4 text-blue-600" />
      <AlertDescription>
        [Dev Mode Simulation] OTP for verification is: <strong>{otpCode}</strong>
      </AlertDescription>
    </Alert>
  )}
  ```

### B. Disable Mock SMS Provider Logging
If you decide to keep the development provider file as a local testing utility but want to ensure it is never triggered in production:
* **File:** `lib/otp/smsProvider.ts`
* **Refactor:** Ensure that the fallback check immediately throws in production mode if real keys are missing:
  ```typescript
  if (process.env.NODE_ENV === 'production' && !process.env.SMS_API_KEY) {
    throw new Error('Production SMS API key is not configured.');
  }
  ```

## 3. Files to Retain Intact (Do Not Delete)

To maintain a clean decoupled architecture, keep the following core modules. They have been written to automatically adapt between development mock flows and real production integrations based solely on the environment config:

1. **`lib/digilocker/sanitiser.ts`**  
   Handles AES-256-GCM encryption of raw Aadhaar UIDs and standardizes all demographic fields (DOB, Gender, Address, ABHA ID).
2. **`lib/digilocker/client.ts`**  
   Performs the secure server-to-server OAuth code exchange, handles XML eAADHAAR parsing, and scans issued documents for linked ABHA accounts.
3. **`lib/digilocker/sessionStore.ts`**  
   Manages server-side transient session records in Firestore with automated TTL eviction to protect PKCE states.
4. **`app/api/digilocker/callback/route.ts`**  
   Acts as the secure boundary proxy that keeps OAuth client secrets and credentials hidden from the client browser.
5. **`lib/providers/digilockerProvider.ts`**  
   Implements the polymorphic `IdentityProvider` contract for the real OAuth popup flow.
6. **`lib/providers/mockProvider.ts`**  
   Retain this for local development and offline mock runs.

## 4. Production Security Checklist

Prior to launching the live DigiLocker integration, verify:
- [ ] **HTTPS Transport:** Verify that all routes (especially API callbacks) redirect HTTP to HTTPS with HSTS enabled.
- [ ] **AES Key Length:** Confirm that `DIGILOCKER_AADHAAR_KEY` is a full 64-character hex key to ensure `aes-256-gcm` executes with the proper key size.
- [ ] **OAuth Origins:** Ensure only your production web domain is white-listed under the "Allowed Callback URLs" in the MeitY DigiLocker Partner Portal.
- [ ] **Firestore Indexing:** Deploy the required indices for `verificationLogs` and `patients` to handle parallel deduplication lookups smoothly.
